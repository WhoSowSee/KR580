# kr-580 — глубокий аудит и рекомендации

Дата: 2026-05-18
Базис: статический разбор `crates/core/src/**/*.rs` + динамика на 16 `.580`-тестах
из `D:/kr/Examples/bug-tests/` и 7 синтетических BUG-репродьюсерах.

Все 55 тестов рабочего пространства зелёные. Триплет S-багов оригинального
`KP580.exe` (BUG-01/08/09) и BUG-02/03/04/05/07 структурно невозможны:
условия читаются через `Cpu8080State::condition()` напрямую из `flags.sign`,
без UI-индирекции. Нижеследующие пункты — не дефекты «сейчас сломано», а
точки, в которых стоит принять решение или закрыть edge-case.

## 1. Неопкоды → `DecodeError` вместо NOP-эквивалента

### Где
`crates/core/src/decode.rs:15`:
```rust
pub const fn is_undocumented_opcode(opcode: u8) -> bool {
    matches!(
        opcode,
        0x08 | 0x10 | 0x18 | 0x20 | 0x28 | 0x30 | 0x38 | 0xCB | 0xD9 | 0xDD | 0xED | 0xFD
    )
}
```
И в `decode_opcode` сразу `return Err(DecodeError::UndocumentedOpcode(opcode))`.
Тест `undocumented_opcode_returns_decode_error_and_does_not_advance_pc`
фиксирует именно такое поведение.

### Что говорит спецификация
На реальном Intel 8080 эти 12 байт не вызывают исключения. Их фактическое
поведение:

| Опкоды | Эквивалент на 8080 |
|---|---|
| `0x08, 0x10, 0x18, 0x20, 0x28, 0x30, 0x38` | `NOP` (4 такта) |
| `0xCB` | `JMP a16` (10 тактов, читает 2 байта операнда) |
| `0xD9` | `RET` (10 тактов) |
| `0xDD, 0xED, 0xFD` | `CALL a16` (17 тактов, читает 2 байта) |

В документации `kr-580 → KP580` (`07-bugs-and-fixes.md`, BUG-06) эта тема
обозначена явно: рекомендация — трактовать неопкоды как функциональные
дубли, чтобы любая 8080-программа исполнялась без падения.

### Вердикт
Не баг, а политика. Текущее поведение — «строгий режим», полезен для отлова
кривых программ и отладки. Но для совместимости с реальным 8080-софтом
эмулятор должен уметь работать в «совместимом» режиме.

### Рекомендация
Сделать переключатель политики на уровне `Cpu8080State` или передавать его
в `step_instruction`/`run_until_halt`:

```rust
pub enum UndocumentedOpcodePolicy {
    Strict,   // вернуть DecodeError (по умолчанию для тестов)
    Compat8080, // выполнить как функциональный дубль
}
```
В compat-режиме `0x08..0x38 (7 байт) → NOP`, `0xCB → JMP`, `0xD9 → RET`,
`0xDD/ED/FD → CALL`. Тайминги по таблице выше.

Минимальный объём правки — две строки в `is_undocumented_opcode`/`decode_opcode`
и один новый бранч в `execute_opcode`. Существующий тест переименовать в
`undocumented_opcode_returns_decode_error_in_strict_mode` и добавить парный
для compat.

## 2. Тайминги Jcc — единый путь 10 тактов

### Где
`decode.rs:142`:
```rust
"J{} a16", ..., InstructionTiming::conditional(10, 10)
```
И `state.rs:97`: `self.cycle_count += u64::from(outcome.t_states)` — для Jcc
`outcome.t_states = info.timing.t_states_taken = 10`, ветвление по
`taken/not_taken` в самом control.rs не учитывается (он не возвращает
другое значение для Jcc, только для Rcc/Ccc).

### Что говорит спецификация
8080 Programmer's Manual, p. 2-7: `Jcc = 10/10` (одинаковое число тактов
независимо от взятия). Документация `branch.md` подтверждает.

### Вердикт
Корректно. Используется `conditional(10, 10)` ровно потому, что
формально это «условный переход», но обе ветки одинаковы. `for_branch`
из `timing.rs:25` тоже отдаёт 10 в обоих случаях.

### Рекомендация
Никаких правок. Если хочется убрать визуальный шум, можно заменить
`conditional(10, 10)` на `fixed(10)` для Jcc — семантика идентична,
а намерение читается лучше. Это косметика.

## 3. Тайминги Rcc — 11/5, не 10/5

### Где
`decode.rs:133`: `R{} ... conditional(11, 5)`
`control.rs:30`: `if taken { 11 } else { 5 }` — runtime тоже даёт 11/5.

### Что говорит спецификация
8080 Programmer's Manual: **Rcc = 11 taken / 5 not-taken**, безусловный
RET = 10 тактов. У kr-580 в `decode.rs:200`: `RET ... fixed(10)` — корректно.

### Расхождение: ошибка в нашей же документации
`D:/kr/docs/03-instructions/branch.md:268`:
```
| `RET` / `Rcc` | 10 / 5 |
```
Это объединение двух команд в одну строку даёт неверное впечатление, что
Rcc = 10/5. Правильно: `RET = 10`, `Rcc = 11/5`.

### Вердикт
Код kr-580 — корректный (11/5). Документация — некорректная.

### Рекомендация
Исправить `D:/kr/docs/03-instructions/branch.md`:
```
| `RET` | 10 |
| `Rcc` (taken / not-taken) | 11 / 5 |
| `CALL` | 17 |
| `Ccc` (taken / not-taken) | 17 / 11 |
| `JMP` / `Jcc` | 10 / 10 |
| `RST n` | 11 |
| `PCHL` | 5 |
```

## 4. SUB AC — формула с `wrapping_add` хрупкая на edge-cases

### Где
`crates/core/src/ops/alu.rs:100`:
```rust
self.flags.auxiliary_carry = (a & 0x0F) >= ((value & 0x0F).wrapping_add(borrow_in));
```

### Анализ
Эталон 8080: `AC = (нет заёма из бита 4)` при вычитании, то есть
`AC = ((a & 0xF) − (b & 0xF) − borrow) ≥ 0`.

Текущая формула делает `(value & 0x0F).wrapping_add(borrow_in)` в `u8`.
Edge case: `value & 0x0F == 0x0F`, `borrow_in == 1`.

```
0x0F.wrapping_add(1) = 0x10  (без переполнения, влезает в u8)
(a & 0x0F) >= 0x10  →  всегда false  (т.к. a & 0x0F ∈ [0..0x0F])
```
Что должно быть математически:
`(a & 0xF) − 0x0F − 1 = (a & 0xF) − 0x10` — всегда отрицательно для
любого `a & 0xF ∈ [0..0xF]`, значит заём всегда есть, AC=0. **Совпадает
с тем, что выдаёт текущая формула. Случайно правильно.**

Но это работает потому, что `0x0F + 1 = 0x10` ещё помещается в `u8` без
обёртки, а `(a & 0x0F) ∈ [0x00..0x0F]`, поэтому `>=` с 0x10 всегда даёт
false. Если когда-нибудь кто-то изменит маски или тип, ловушка сработает.

Тест `subtract_auxiliary_carry_matches_prompt_edge_cases` проходит, но
не покрывает этот конкретный край (случай SBB с `value & 0xF == 0xF` и
CY=1).

### Вердикт
Сейчас работает, но логика опасная. Лучше переписать в `u16` для ясности.

### Рекомендация
```rust
pub(crate) fn sub(&mut self, value: u8, borrow: bool, write_result: bool) {
    let borrow_in = u16::from(borrow && self.flags.carry);
    let a = self.registers.a as u16;
    let v = value as u16;
    let raw = a.wrapping_sub(v).wrapping_sub(borrow_in);
    let result = raw as u8;

    self.flags.carry = a < v + borrow_in;
    // AC = «нет заёма из бита 4»; считаем в полноразрядной арифметике:
    let lo_a = a & 0x0F;
    let lo_b = (v & 0x0F) + borrow_in;
    self.flags.auxiliary_carry = lo_a >= lo_b;

    self.flags.set_sign_zero_parity(result);
    if write_result {
        self.registers.a = result;
    }
}
```
Семантика та же, но без скрытой зависимости от того, что `0x0F + 1` ещё не
обернулось. Добавить тест на SBB с `A=0x10, B=0x0F, CY=1` (ожидаемое:
`A=0x00, AC=0, CY=0`) и `A=0x00, B=0x0F, CY=1` (ожидаемое: `A=0xF0, AC=0,
CY=1`).

## 5. DAA AC — устанавливается всегда, даже если коррекция была 0x60

### Где
`crates/core/src/ops/alu.rs:132-150`:
```rust
let result = a.wrapping_add(correction);
self.flags.auxiliary_carry = ((a & 0x0F) + (correction & 0x0F)) > 0x0F;
```

### Анализ
Здесь `correction & 0x0F` всегда даёт либо `0x06` (если первая ветка
сработала), либо `0x00` (иначе). Если сработала **только** вторая ветка
(`correction = 0x60`), то `correction & 0x0F = 0`, и AC будет `(a & 0x0F) > 0x0F`,
то есть всегда false. Это корректно: при коррекции high-nibble на 0x60
low-nibble не трогается, заёма из бита 4 нет, AC должен быть 0. Совпадает.

Если сработали обе ветки (`correction = 0x66`), `correction & 0x0F = 6`,
AC = `(a & 0x0F) + 6 > 0xF`. Корректно: low-nibble добавил 6, мог быть
заём из бита 3.

Если не сработала ни одна (`correction = 0`), AC = `(a & 0x0F) > 0xF`,
всегда false. **Здесь спорно**: 8080 спецификация говорит, что DAA
устанавливает AC «if a carry out of bit 3 occurred during the addition».
Если коррекции не было, заёма тоже не было, AC=0. Совпадает.

### Вердикт
Алгоритмически правильно, но текущая запись маскирует логику. Добавить
комментарий с явной формулой.

### Рекомендация
Не править код, только добавить регрессионный тест с тремя сценариями:
- `A=0x9A, AC=0, CY=0 → DAA → A=0x00, CY=1, AC=1` (обе коррекции, low-nibble была 0xA, +6=0x10, заём из бита 3)
- `A=0xA0, AC=0, CY=0 → DAA → A=0x00, CY=1, AC=0` (только high, low-nibble не трогалась)
- `A=0x9F, AC=0, CY=0 → DAA → A=0x05` (low: 0xF+6=0x15 → A=0xA5, AC=1; high: 0xA0+0x60=0x05, CY=1) — это особенно полезно для регрессии.

Текущий тест `bug04_daa_adjusts_bcd_addition` проверяет только базовый
кейс `0x15+0x27`. Расширить.

## 7. Прерывания и INTE — реализованы

### Где
`crates/core/src/state.rs:10-15` поля:
```rust
pub interrupt_request_pending: bool,
pub interrupt_enable: bool,
pub interrupt_enable_pending: bool,
```
`execute.rs:5-58`: `can_accept_interrupt`, `accept_interrupt`,
`execute_instruction_boundary` корректно обрабатывает:
- проверку прерывания на границе инструкции;
- отложенный EI (включается **после** следующей инструкции);
- DI выключает оба флага сразу;
- на принятии прерывания: `IE := false`, `halted := false`, push PC,
  jmp на `(vector >> 3) * 8`.

### Анализ
Эталон 8080:
- INT принимается только на границе инструкции при IE=1;
- EI должен включить IE только после следующей инструкции (защита первой
  инструкции после EI от прерывания) — реализовано через
  `interrupt_enable_pending`;
- HLT выходит из останова при INT — реализовано (`halted := false` в
  `accept_interrupt`).

### Вердикт
Реализовано полностью и корректно.

### Рекомендация
Добавить тесты:
- INT во время HLT возобновляет исполнение;
- EI + (одна инструкция) + INT срабатывает на следующей границе;
- EI + INT (на той же границе) не срабатывает (защита первой инструкции).

Сейчас в `cpu_semantics.rs` есть `push_pop_psw_roundtrips_flags_with_reserved_bits_normalized`
и общие тесты декодера, но именно «прерывания» как сценарий не покрыты —
найдите 1-2 теста на это.

## 8. HLT + interrupt — `run_until_halt` не различает «остановлен навсегда» и «ждёт прерывания»

### Где
`state.rs:142`:
```rust
pub fn run_until_halt<B: PortBus>(
    &mut self, bus: &mut B, max_instructions: u64,
) -> Result<u64, CoreError> {
    let mut executed = 0;
    while !self.halted && executed < max_instructions { ... }
    Ok(executed)
}
```

### Анализ
Цикл прерывается по первому `halted=true`. Если INT придёт в этот момент,
`run_until_halt` уже вышел и прерывание не обрабатывается (хотя при
следующем `step_instruction` обработалось бы).

Это семантически правильно для бенчмарков «прогнать программу до HLT»,
но если прерывания используются (например, periodic timer), цикл должен
дорабатывать.

### Вердикт
Не баг. Полностью соответствует имени «`run_until_halt`».

### Рекомендация
Добавить парный метод:
```rust
pub fn run_until_halt_with_interrupts<B: PortBus>(
    &mut self, bus: &mut B, max_instructions: u64,
) -> Result<u64, CoreError> {
    let mut executed = 0;
    while executed < max_instructions {
        if self.halted && !self.can_accept_interrupt() {
            // полностью остановлены, прерывания невозможны
            break;
        }
        self.step_instruction(bus)?;
        executed += 1;
    }
    Ok(executed)
}
```
Не обязательно — можно просто оставить пользователю ручной цикл по
`step_instruction`.

## 9. `step_tact` — детализация фаз M-cycle

### Где
`state.rs:101-124`:
```rust
pub fn step_tact<B: PortBus>(&mut self, bus: &mut B) -> Result<TactOutcome, CoreError> {
    if self.active_tacts_remaining == 0 {
        ...
        self.active_tacts_total = t_states;
        self.active_tacts_remaining = t_states;
        ...
    }
    let phase = self.active_tacts_total - self.active_tacts_remaining;
    self.active_tacts_remaining -= 1;
    ...
}
```

### Анализ
Текущая модель: «инструкция выполняется атомарно при первом такте, потом
N−1 тактов «пустого ожидания» до следующей границы». Эта модель совпадает
по числу тактов с эталоном, но **не моделирует** машинные циклы (M1, M2,
M3, ...) и фазы T1..T5 внутри каждого M-цикла, в отличие от того, что
делает оригинальный «обучающий» KP580.exe.

Для прогона программ это эквивалентно. Для **обучающего режима**
(подсветка фаз шины, индикаторы M-цикла, состояние `WR`/`DBIN`) такая
модель недостаточна.

### Вердикт
Точность по T-states — корректная. Точность по фазам шины — отсутствует.

### Рекомендация
Если планируешь воспроизводить «обучающий» интерфейс KP580 с лампочками
M-цикла:
1. Расширить `InstructionTiming` полем `machine_cycles: Vec<MachineCycle>`,
   где `MachineCycle` описывает тип (M1=opcode fetch, M2/M3=memory read/write,
   I/O read/write, ...) и число тактов (3..5).
2. В `step_tact` отдавать в `TactOutcome` номер машинного цикла и его тип.
3. Покрыть стандартными циклами: MOV r,r = 1×M (4T fetch+1 padding); MOV r,M
   = 2×M (4T+3T); CALL = 5×M (5T+3T+3T+3T+3T) и т.д.

Это большая работа, нужна только если делаешь именно «обучающий» эмулятор.
Для прогона ROM/программ — не нужно.

## 13. AC у INR/DCR — формула расходится с реальной арматурой 8080

### Где
`alu.rs:72`: `INR: aux = (value & 0x0F) == 0x0F`
`alu.rs:79`: `DCR: aux = (result & 0x0F) != 0x0F`

### Анализ
- INR: AC=1 если low-nibble стал 0 (т.е. был 0xF). Корректно.
- DCR: после `result = value − 1`, если `value & 0x0F == 0`, то
  `result & 0x0F == 0xF` — был заём. AC=1 если заём = `(result & 0x0F) != 0x0F` — это **наоборот**.

Подождите. У 8080 AC после DCR — это «нет заёма из бита 4», т.е.
**AC=1 если заёма не было**. Значит:
- `value & 0x0F == 0x00` → `result & 0x0F == 0x0F` → заём был → AC=0
- `value & 0x0F != 0x00` → заёма не было → AC=1

`(result & 0x0F) != 0x0F` даёт ровно это:
- если result low-nibble = 0xF (был заём) → false → AC=0 ✓
- если result low-nibble != 0xF (заёма не было) → true → AC=1 ✓

### Вердикт
Корректно. Я запутался при первом чтении формулы. Тест
`subtract_auxiliary_carry_matches_prompt_edge_cases` это покрывает.

### Рекомендация
Добавить комментарий к строке:

## 16. Декодер использует общие шаблоны без таблицы — риск опечатки

### Где
`decode.rs` использует битовые шаблоны (`opcode & 0xC7 == 0xC0` и т.п.)
вместо явной таблицы 256 опкодов.

### Анализ
Подход чистый и компактный, но менее устойчив к опечаткам, чем явная
таблица. Существующий тест `every_documented_opcode_executes_from_controlled_state`
гоняет все 256 опкодов и проверяет, что каждый либо выполняется, либо
возвращает `UndocumentedOpcode`. Это надёжный регрессионный страхователь.

### Вердикт
Архитектурно нормально, тесты закрывают риск опечатки.

### Рекомендация
Добавить отдельный тест, который сравнивает результат `decode_opcode(N)`
для всех 256 N с захардкоженной табличкой `(mnemonic, size, taken_t)`.
Если файл такой таблицы соберёшь из 8080 Programmer's Manual — это будет
канонический регрессионный лист.
